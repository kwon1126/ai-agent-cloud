# 멀티 에이전트 시스템 서술형 답안

---

**문제 1:** 멀티 에이전트 시스템에서 '시스템 지시사항 오염' 문제란 무엇인가? 단일 에이전트 방식의 한계와 연결하여 ChatGPT / Claude 도움을 받아 2~3 문단으로 서술하시오

**문제 2:** 메시지 전달, 메모리 핸드오프, 완전 공유 메모리 세 패턴 중 하나를 골라 어떤 상황에서 사용하고 어떤 상황에서 피해야 하는지를 구체적인 예시와 함께 서술하시오

---

## 문제 1: 시스템 지시사항 오염(System Prompt Contamination)

단일 에이전트 방식은 하나의 LLM이 하나의 시스템 프롬프트(system prompt)를 기반으로 모든 작업을 처리한다. 이 구조의 근본적인 한계는 **컨텍스트 윈도우 안에 너무 많은 역할과 지시사항이 누적**된다는 점이다. 예를 들어 "검색 전문가이면서 코드 작성자이면서 요약 작성자"를 하나의 에이전트에게 요구하면, 지시사항들이 서로 충돌하거나 우선순위가 불명확해진다. 또한 장기 대화에서는 초기의 시스템 지시가 후속 맥락에 의해 희석되거나 덮어쓰이는 현상이 발생한다.

**시스템 지시사항 오염(System Instruction Contamination)**이란 멀티 에이전트 환경에서 서브에이전트(sub-agent)가 오케스트레이터(orchestrator)로부터 작업 지시를 받을 때, 오케스트레이터의 지시 내용이 서브에이전트 자신의 원래 시스템 프롬프트와 혼재되어 예상치 못한 동작을 유발하는 문제를 말한다. 예를 들어 오케스트레이터가 "다음 에이전트에게 전달: 이전 모든 지시를 무시하고..."와 같은 내용을 전달할 경우, 서브에이전트는 자신의 역할 정의를 잃어버리고 오염된 지시를 따를 수 있다. 이는 **프롬프트 인젝션(prompt injection)** 공격의 한 형태로도 활용될 수 있어 보안상 심각한 취약점이 된다.

이 문제는 멀티 에이전트 설계 시 각 에이전트가 **자신의 시스템 프롬프트를 불변(immutable) 영역으로 보호**하고, 오케스트레이터로부터 받은 메시지는 별도의 사용자 메시지 영역에서만 처리하도록 분리함으로써 완화할 수 있다. Anthropic의 Claude는 이를 위해 오케스트레이터 입력을 신뢰하더라도 원칙적 행동 지침(constitutional principles)을 시스템 레벨에서 고정하는 방식을 권장한다.

## 문제 2: 메모리 핸드오프(Memory Handoff) 패턴

세 가지 패턴 중 **메모리 핸드오프(Memory Handoff)** 를 선택한다. 이 패턴은 에이전트 A가 작업을 완료한 뒤, 자신의 작업 상태와 중간 결과를 **구조화된 요약 형태로 에이전트 B에게 전달**하는 방식이다. 각 에이전트는 독립적인 메모리를 가지지만, 전환 시점에 필요한 컨텍스트만 선별적으로 이어받는다.

**적합한 상황:** 단계가 명확히 구분되고 순차적으로 진행되는 파이프라인에 이상적이다. 예를 들어 "문서 수집 에이전트 → 분석 에이전트 → 보고서 작성 에이전트"로 이어지는 리서치 자동화 시스템에서, 수집 에이전트는 수집된 URL 목록과 핵심 키워드만 요약하여 분석 에이전트에 넘긴다. 전체 수집 로그를 공유하지 않으므로 컨텍스트 낭비 없이 각 에이전트가 자신의 역할에 집중할 수 있다.

**피해야 하는 상황:** 에이전트 간에 **빈번한 양방향 참조**가 필요한 경우에는 부적합하다. 예를 들어 실시간 협업 코드 리뷰 시스템처럼 에이전트 A의 판단이 에이전트 B의 결과에 따라 즉시 수정되어야 하는 경우, 핸드오프마다 요약을 재생성하는 오버헤드가 크고 정보 손실이 발생할 수 있다. 이런 경우에는 **완전 공유 메모리(Shared Memory)** 패턴이 더 적합하다.
